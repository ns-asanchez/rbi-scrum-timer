🖥️ Local Claude Setup → Repositories

  • Claude Code CLI on AWS Bedrock — no direct Anthropic API cost, integrated with team AWS profile
  • Central CLAUDE.md as global index: 5 repos, 20+ skills, team rules, label conventions
  • Skills distributed by domain — Jira ops, PR review, oncall, infra — invoked with /skill-name from anywhere
  • Persistent memory system (user / project / feedback) — Claude remembers decisions & preferences across sessions

🎯 How the Need Was Detected

The Pain

  • No time control — someone always overran
  • Checking tasks required switching to Jira, breaking flow
  • No speaking time history or trends
  • Existing solution: phone timer + Jira on a 2nd screen

The Outcome

  • Single macOS app: timer + live Jira context
  • Sprint tasks per speaker (Active + Done)
  • Session history with charts
  • Team member sync from Jira board
Estimated effort without AI: ~40h · Delivered with AI: ~6h

⚙️ The Process

Need → Use case → Development → Quality → Lessons learned

  • Need: one desktop tool combining timer + live Jira sprint context
  • Use case: active speaker sees their sprint tasks; team sees who's talking and for how long
  • Development: strictly iterative — timer → Jira → avatars → sprint info → QA fields → type badges, one commit per feature
  • Quality: pylint 10/10, black + isort, docstrings on all methods, .gitignore for DB & credentials
  • Interaction: Alex provided exact context (field IDs, screenshots, logs, DNS traces) → Claude implemented. Being specific = cost control. Every precise message replaced 3–5 exploratory iterations.

⚙️ Key Lessons Learned

Jira API

  • Custom field IDs are instance-specific
  • /search → /search/jql (migration)
  • Tenant DNS uses ECS subnet routing
  • greenhopper needed for sprint reports

SQLite / macOS / Tk

  • Design for migrations from day one
  • CTkImage must be cached — reconfiguring causes crash
  • Tk9 scroll: unsigned 16-bit deltas, not standard
  • transient() for dialogs on macOS

⚠️ Limitations — Process & Interaction

  • Context compression: Long sessions compress early decisions — DB schema changes, layout weights can be lost mid-session
  • No cross-session memory by default: Without CLAUDE.md + memory files, every session starts blank
  • Precision is a cost driver: Vague inputs → exploratory loops. Exact IDs, logs, URLs → 1 iteration instead of 5
  • UI iteration is expensive: implement → launch → screenshot → describe → fix. No headless UI testing available
  • Sequential approval: Each file write needs human approval — latency for rapid UI work (mitigated via settings.json)
  • No rollback awareness: Claude doesn't track git history — regression debugging was purely conversational
  • Single-file context: Cross-file refactors need explicit instruction; no proactive architecture suggestions

🤔 What I Would Have Done Differently

Claude's honest reflection on what slowed us down

  • Scroll fix: macOS Tk9 unsigned 16-bit deltas — researched from scratch multiple times. A single upfront test would have solved it once.
  • Dialog topmost: 4–5 iterations before finding transient(). Knowing the macOS constraint upfront = 1 iteration.
  • Jira field IDs: Discovered customfield_10004 only when customfield_10016 returned empty. A 5-min field audit at the start saves multiple debug cycles.
  • Layout breaks: Happened twice from editing without reading the full current file state. "Always read before edit" discipline would have prevented regressions.
Main lever: front-load specification — screen layout sketch, Jira field inventory, platform constraints checklist — before writing the first line of code.

✅ Is This Viable for Netskope Feature Development?

YES Works great for

  • Internal tooling & automation
  • QE scripts & log analysis
  • Ticket refinement & PR review
  • Greenfield tools with clear scope

GUARDRAILS Needs oversight for

  • Production code
  • Architecture decisions
  • Cross-team dependencies
  • Security-sensitive changes
The real multiplier: not that Claude writes code faster — it eliminates context-switch cost. Alex stayed in flow; Claude handled implementation details. ~6x productivity gain on greenfield internal tooling.

✅ Knowledge Sources That Work

  • Confluence as context — architecture docs, team conventions, sprint pages briefed into the session
  • Jira as spec — ticket description = feature requirements, AC = acceptance criteria
  • GitHub repos as code reference — existing patterns, API clients, test helpers
  • Logs & screenshots as debugging input — DNS traces, Jenkins output, UI screenshots
Brief Claude like a senior engineer joining the team — give the context you'd give a person. The quality of output is directly proportional to the quality of context.

📋 Spec-Driven Frameworks

aispec.org · Augment Code

What they offer

  • aispec.org: PRD → TDD → implementation before any code
  • Augment Code: codebase-awareness tooling for existing large repos
  • Reduce mid-development direction changes

Honest verdict

  • For this project: conversational worked better — features were small, iterative, tight feedback loop
  • For a large Netskope production feature: spec-driven pays for itself in alignment
  • Value scales with feature size & team size
Solo dev on internal tooling → overhead outweighs benefit.
Multi-engineer feature with stakeholders → spec reduces back-and-forth significantly.

💡 Conclusion

AI pair programming is real — but it requires the same discipline as human pair programming: clear communication, shared context, and defined responsibilities.

Claude is most effective when treated like a senior engineer on their first week: capable of executing at a high level, but dependent on you to provide domain knowledge, constraints, and judgment calls.

The RBI Scrum Timer went from idea to a polished, git-tagged desktop app in a single session. The team has a tool that didn't exist before, at a fraction of the traditional cost — and the process itself is a reusable playbook for AI-assisted development at Netskope.
